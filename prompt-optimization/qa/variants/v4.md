**Mission:** Generate authoritative, detailed, and production-ready Q&A training pairs from code scan data. These pairs will be used to fine-tune Gemma 4 for code review and quality analysis.

## Core Principles

1. **Depth over breadth:** Provide comprehensive analysis, not superficial checklists
2. **Actionable insights:** Every finding should include concrete fixes, not just identification
3. **Contextual awareness:** Consider language idioms, framework conventions, and real-world constraints
4. **Pedagogical value:** Explain the "why" behind recommendations, not just the "what"

## Output Standards

### Format Requirements
- **JSONL format** with one object per line
- **No trailing commas** or invalid JSON
- **Proper escaping** of quotes and special characters
- **Consistent indentation** (2 spaces for readability)
- **Language-specific terminology** (use correct terms for JavaScript, Go, Python, etc.)

### Quality Benchmarks

#### Security Reviews Should Include:
- **Threat modeling** context (what could an attacker do?)
- **CVSS scoring** or equivalent severity assessment
- **Exploit scenarios** (how could this be triggered in production?)
- **Defense-in-depth** recommendations (multiple layers of protection)
- **Code examples** showing both vulnerable and secure patterns
- **Reference standards** (OWASP Top 10, CWE, etc.)

#### Production Readiness Analysis Should Include:
- **Root cause analysis** (why is this a problem?)
- **Impact assessment** (what happens if it fails?)
- **Monitoring implications** (how would you detect issues?)
- **Performance considerations** (any hidden costs?)
- **Scalability limits** (when does it break?)
- **Testing gaps** (what tests would catch this?)

#### Fix Recommendations Should Include:
- **Step-by-step implementation** (exactly what to change)
- **Alternative approaches** with trade-offs
- **Backward compatibility** considerations
- **Testing strategy** (how to verify the fix?)
- **Performance impact** analysis
- **Migration path** if breaking changes

## Generation Process

### Step 0: DATA QUALITY GATE ⛔ (check BEFORE generating anything)

For each scan entry, verify ALL of the following are non-empty:
- Issue type / category (e.g., "SQL_INJECTION", "NULL_DEREFERENCE")
- Code snippet or file context (the actual code being reviewed)
- Severity level (LOW / MEDIUM / HIGH / CRITICAL)

**If ANY of these are missing or blank → SKIP this entry entirely. Do not output a JSON line for it.**

❌ Bad scan entry (skip this):
```
File: 
Line: 
Severity: 
Issue type: 
```

✅ Good scan entry (process this):
```
File: src/auth/jwt.ts
Line: 42
Severity: HIGH
Issue type: HARDCODED_SECRET
Code: const secret = 'abc123';
```

Incomplete entries produce useless training data. Quality over quantity — 10 good pairs beat 50 empty ones.

### Step 0b: CODE EXAMPLE GATE ⛔ (check AFTER drafting your output, BEFORE writing it)

Every output MUST contain at least one before/after code example showing the problematic code and the fix. No exceptions.

**If you cannot write a meaningful before/after code snippet for this issue → SKIP the entry.**

### Step 1: Scan Analysis
For each issue in the scan JSON (that passed the quality gate):
- **Categorize** by type (security, performance, maintainability, etc.)
- **Prioritize** by impact and likelihood
- **Contextualize** within the codebase and framework
- **Research** any unfamiliar patterns or libraries

### Step 2: Question Formulation — ISSUE-SPECIFIC INSTRUCTIONS (NEW IN v4)

**The golden rule:** Every instruction must name the specific issue, API, pattern, or symptom found — never a generic category prompt.

**❌ BANNED — generic category prompts:**
```
"instruction": "Analyze this TypeScript code for security vulnerabilities. Consider: authentication, authorization, input validation..."
"instruction": "What production-readiness problems exist in this Go code?"
"instruction": "Review this code for security issues: [raw scan text]"
```

**✅ REQUIRED — issue-embedded prompts:**
```
"instruction": "How does using Object.defineProperty without key validation enable prototype pollution in Vue 2's reactivity observer?"
"instruction": "Why does ioutil.NopCloser trigger static analysis warnings in Go 1.16+, and how do you migrate to io.NopCloser?"
"instruction": "How can missing retry logic for 5xx responses in go-kit HTTP clients cause cascading failures under load?"
"instruction": "What makes time.sleep(1) in yt-dlp's download loop uncancellable, and how would you replace it with interruptible sleep?"
```

**Formula for a good instruction:**
`How does [specific_pattern/API/symptom] [cause/enable/trigger] [specific_problem] in [language/framework/context]?`

Or for fix questions:
`Why does [specific deprecated/broken thing] [behave this way] and how do you [specific migration/fix]?`

**Generate 2-3 distinct question angles per issue:**
- **Angle 1:** Mechanism question — how/why does this specific pattern cause the problem?
- **Angle 2:** Impact question — what breaks in production when this specific issue triggers?
- **Angle 3:** Fix question — what's the idiomatic [language] way to fix [specific pattern]?

### Step 3: Answer Generation
Each answer must be:

**Length:** 150-500 words (substantial but focused)
**Structure:**
1. **Summary** (1-2 sentences naming the specific issue)
2. **Detailed analysis** (break down the problem)
3. **Impact assessment** (what happens if not fixed?)
4. **Multiple solution options** (pros/cons)
5. **Implementation guidance** (before/after code examples — REQUIRED)
6. **Testing/validation** (how to verify the fix)
7. **References** (standards, documentation, best practices)

**Tone:** Authoritative, educational, collaborative
**Language:** Professional but accessible (avoid unnecessary jargon)

### Good Issue-Embedded Security Question:
```json

  "instruction": "How does the hardcoded JWT secret 'abc123' in jwt.ts allow token forgery without any network access?",
  "output": "CRITICAL HARDCODED_SECRET: A static secret like 'abc123' embedded in source code means any developer with repo access — or any attacker who obtains the source — can mint arbitrary valid JWTs. **CVSS: 9.0 (Critical)**. Unlike key theft from a running system, this requires zero network access. **Exploit:** `jwt.sign({userId: 1, role: 'admin'}, 'abc123')` forges an admin token instantly. **Fix:**\n\n```javascript\n// BEFORE: anyone with git access can forge tokens\nconst jwt = require('jwt-simple');\nconst secret = 'abc123';\n\n// AFTER: secret lives only in the environment\nconst jwt = require('jsonwebtoken');\nconst secret = process.env.JWT_SECRET; // set in deployment env, not code\nif (!secret) throw new Error('JWT_SECRET env var is required');\n```\n\nAlso rotate to RS256 so the signing key never leaves the auth service. **Test:** Deploy without JWT_SECRET set — the startup check should throw. **Ref:** OWASP A02:2021, CWE-321."

```

### Good Issue-Embedded Production Question:
```json
{
  "instruction": "Why does starting an HTTP server without context cancellation cause request failures during Kubernetes rolling deploys?",
  "output": "MEDIUM NO_GRACEFUL_SHUTDOWN: When a Go HTTP server starts with `http.ListenAndServe` but no signal handling, SIGTERM from Kubernetes during a rolling deploy immediately kills the process, dropping all in-flight requests. **Impact:** Every deploy causes a spike of 500s equal to the number of requests in-flight at shutdown time. **Root cause:** `ListenAndServe` blocks forever with no cancellation path. **Fix:**\n\n```go\n// BEFORE: SIGTERM kills in-flight requests instantly\nfunc main() {\n    http.ListenAndServe(':8080', handler)\n}\n\n// AFTER: waits up to 30s for in-flight requests to finish\nfunc main() {\n    srv := &http.Server{Addr: ':8080', Handler: handler}\n    ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, os.Interrupt)\n    defer stop()\n    go srv.ListenAndServe()\n    <-ctx.Done()\n    shutCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second)\n    defer cancel()\n    srv.Shutdown(shutCtx)\n}\n```\n\n**Test:** `kubectl rollout restart` while load testing — verify zero 5xx during the deploy. Match Kubernetes' `terminationGracePeriodSeconds` to your timeout."
}
```

## Common Pitfalls to Avoid

- ❌ Generating pairs from incomplete scan entries (missing issue type, code, or severity)
- ❌ Outputs without code examples — **skip any output you can't show with before/after code**
- ❌ **Generic category instructions** ("Analyze this code for X") — name the specific issue
- ❌ Vague statements ("this is insecure" without explanation)
- ❌ No severity assessment (critical vs minor)
- ❌ One-size-fits-all solutions (consider context)
- ❌ Ignoring framework conventions (Go's error handling, JS async patterns)
- ❌ Forgetting testing/validation steps
- ❌ Overly long answers (>500 words) without clear structure
