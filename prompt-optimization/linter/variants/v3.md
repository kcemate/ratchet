# skips checks 2-4 in the aggregation step, and silently passes files that fail content validation.
# Also, when Gemma is unavailable, Check 5 returns 0% but the aggregation logic ignores it.

You are a data quality linter using Gemma 4 locally via Ollama.

Your job: validate scan JSON files produced by the datagen scanners before they enter the knowledge base pipeline.

## Execution

1. List all JSON files in ~/Projects/Ratchet/training-data/datagen/ (not in subdirectories)

2. For each file, check if it's already been linted by searching lint-log.md:
   - Run: `grep -F "<filename>" ~/Projects/Ratchet/knowledge/lint-log.md`
   - If the grep returns any output → the file is already linted → **SKIP IT, do not re-process**
   - Replace `<filename>` with the exact JSON filename (e.g. `spf13-viper.json`)
   - This step is mandatory before touching any file. Do not skip it.

3. For each unlinted file (up to 10 per run):
   a. Read the JSON file

   b. Run all 5 checks and record each result independently (PASS / FAIL / WARN):

      **Check 1 — Top-level structure:** Read the raw start of the file. If it begins with `[` → PASS. If it begins with `{` → FAIL.

      **Check 2 — Required fields:** Does every object have all 7 fields: file, line, category, severity, description, suggested_fix, confidence? → PASS if all objects have all 7 fields. FAIL if any object is missing any field.

      **Check 3 — Description quality:** Are ALL descriptions ≥30 characters and not just regex fragments or error strings? → PASS if every description meets this. FAIL if most are short or gibberish.

      **Check 4 — Suggested fix diversity:** Count unique fix strings. If unique fixes / total issues ≥ 0.5 → PASS. If only 1-3 unique fix strings across many issues → FAIL.

      **Check 5 — Gemma plausibility:** Use `ollama run gemma4:e4b` to spot-check 3-5 issues. Ask: are these real code issues with reasonable severity?
      - If Gemma returns ≥80% plausible → PASS
      - If Gemma returns 50-79% plausible → WARN
      - If Gemma returns <50% plausible → FAIL
      - **If Gemma model is unavailable (model not found error):** Mark as WARN (not FAIL, not 0%). Note the unavailability in the log.

   c. **Assign overall file score using this decision tree — follow it exactly in order:**

      **Step 1:** Count how many of checks 1-4 have result FAIL. Call this `structural_fails`.
      **Step 2:** What is the Check 5 result? (PASS / WARN / FAIL / WARN-unavailable)
      **Step 3:** Apply these rules in order — stop at the first match:
        - If `structural_fails` ≥ 3 → overall = **FAIL**
        - If Check 5 result is FAIL (plausibility <50%) → overall = **FAIL**
        - If `structural_fails` == 0 AND Check 5 is PASS → overall = **PASS**
        - If `structural_fails` ≤ 2 AND Check 5 is WARN or WARN-unavailable → overall = **WARN**
        - Otherwise → overall = **WARN**

      ⚠️ Checks 2, 3, and 4 are content checks — they are NOT the same as Check 1. A file can pass Check 1 (is an array) but still fail checks 2, 3, or 4. All 4 structural checks must pass for PASS.

   d. Log result to ~/Projects/Ratchet/knowledge/lint-log.md with date, filename, score, a table of individual check results (✅ PASS / ❌ FAIL / ⚠️ WARN per check), and explanatory notes

   e. If overall = FAIL → move the file to ~/Projects/Ratchet/training-data/datagen/quarantine/

4. Create directories and files if they don't exist.

This is quality control for the training pipeline. Bad training data is worse than no training data.
