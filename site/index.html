What is a click?

Every click is one autonomous improvement cycle. Ratchet analyzes your code, picks the highest-impact change, builds it, runs your tests, and commits — all without human intervention. Each click locks in progress. The ratchet only turns one way.

1

Analyze

Scan the target area. Identify the highest-impact improvement.

2

Propose

Pick ONE specific change. Not a laundry list — a focused fix.

3

Build

Write the code. Real files, real changes.

4

Test

Run the full test suite. Green or it doesn't ship.

5

Commit

Lock it in. Descriptive message, clean history.

6

Repeat

Next click. Keep going until the socket is tight.

🛡️ The Pawl

If tests fail, nothing commits. The ratchet can't slip backward — your codebase can only get better, never worse. That's the Pawl.

Boundaries with .ratchet.yml

Your team's architectural intent, codified. Tell Ratchet what to improve — and what to leave alone.

agent: default
defaults:
  clicks: 7
  test_command: npm test
targets:
  - name: error-handling
    path: src/api/
    description: "Improve API error handling"
boundaries:
  - path: src/auth/
    rule: no-modify
    reason: "Auth architecture is intentional" # The team decides

Common questions

Does it work on any codebase?

Yes. Ratchet is language-agnostic. If you can define a test command that exits 0 on green, Ratchet can ratchet. It's been run on Node.js, Python, Go, and Ruby codebases.

What if the AI breaks something?

That's exactly what the Pawl is for. If tests don't pass, nothing commits. The branch state is restored. Your main branch can only receive changes that pass your test suite — structurally, not by convention.

How is this different from running an AI coding agent directly?

An AI coding agent gives you one shot at a task. Ratchet is the loop — it orchestrates sustained iteration: analyze → propose → build → test → commit → repeat. Each click verifies before committing. You get compounding improvement, not a single pass.

Can I scope it to only touch certain files?

Yes — that's what .ratchet.yml is for. Set paths for targets (what to improve) and boundaries (what to never touch). Your auth module stays locked. Your error handling gets tighter.

How many clicks should I run?

Start with 3–5 to get a feel for it. Production runs are typically 7–14. Each click is one focused change, so 7 clicks means 7 committed, tested improvements — not 7 all-in-one rewrites.
