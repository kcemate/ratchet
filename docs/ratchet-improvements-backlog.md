# Ratchet Improvements Backlog
_Generated: 2026-03-22 from dogfooding ratchet-oss_

## Critical — Learned from Eating Our Own Dog Food

### 1. Score-Weighted Click Priority (tk_71wx)
**Problem:** Agent spent 11 consecutive clicks on dedup refactors in a maxed-out category (85.5→85.5). It picks the highest issue COUNT, not the highest score IMPACT.
**Fix:** `priority = (maxScore - currentScore) × fixProbability`. Skip maxed categories entirely. If Testing is 19/25 and Code Quality is 15/15, the agent should never touch Code Quality.
**Impact:** Would have saved ~$15 in API costs and 90 minutes of wasted clicks.

### 2. Stall Detection Upgrade (tk_71wx)
**Problem:** After 2 clicks at ±0, ratchet escalates to "cross-file sweep mode" — but sweep mode just does the same thing with more files. It should switch CATEGORIES, not strategies.
**Fix:** 2 consecutive ±0 clicks → force-switch to highest-headroom category. 3 consecutive ±0 → stop run early, report "score ceiling reached for current strategy."

### 3. Resume Flag (tk_71wx)
**Problem:** Long runs (35-70min) get killed by timeouts, SSH disconnects, CI limits. State is saved to .ratchet-state.json but never read back.
**Fix:** `ratchet torque --resume` reads .ratchet-state.json, skips completed clicks, continues from last checkpoint. Also saves partial PDF progress.

### 4. Click Planner Transparency
**Problem:** No visibility into WHY the agent chose a particular fix. User sees "analyzing..." then gets a dedup refactor when they expected test coverage.
**Fix:** Print the priority ranking before each click: `Click 3: targeting Testing/Coverage (7.5/8, +0.5pts available) — skipping Code Quality (15/15, maxed)`. Let users see the decision.

### 5. Score Regression on Refactors
**Problem:** Clean refactors (extracting shared utils) sometimes INCREASE duplication count temporarily (new shared file + old code before cleanup), causing score regression → rollback → wasted click.
**Fix:** Two-click atomic refactors: extract + cleanup as a single transaction. Or: temporarily exempt the "new shared util" file from duplication scoring during the same run.

### 6. PDF Generation Crash
**Problem:** `TypeError: Cannot read properties of undefined (reading 'catch')` at end of every torque run. PDF report fails silently.
**Fix:** Debug the pdf-report.ts pipeline. Likely a missing async handler or undefined report object.

## Medium — UX Improvements

### 7. --background Mode
**Problem:** Users must keep terminal open for 35-70 minute runs.
**Fix:** `ratchet torque --background` detaches process, writes results to `.ratchet-results.json`, prints "Run started. Check results with `ratchet status`."

### 8. Progress Webhook
**Problem:** CI pipelines need click-by-click status without polling.
**Fix:** `--webhook https://...` POSTs JSON after each click: `{click: 3, score: 87.5, delta: +2, commit: "abc123"}`.

### 9. Per-Click Timeout with Graceful Save
**Problem:** Hard process kill loses in-progress work.
**Fix:** Per-click timeout (configurable, default 10min). On timeout: save state, rollback partial changes, report which click timed out. Don't kill the whole run.

### 10. --focus Flag Should Override Planner
**Problem:** `--focus coverage` was ignored — agent still did dedup refactors. The flag influences prompt text but doesn't constrain the planner.
**Fix:** When --focus is set, ONLY present issues from that category to the agent. Filter the scan results before building the prompt. No ambiguity.

## Low — Nice to Have

### 11. Category Score Ceiling Warning

### 12. Estimated Time Remaining
Based on average click duration, show "~23 minutes remaining (4 clicks × ~6min avg)".

## Lessons for the Score Algorithm

1. **Issue count ≠ importance.** 152 repeated lines sounds bad but scores 3/3. 31 untested files scores 6/8 with 2pts to gain. The planner must understand the difference.
2. **Maxed categories are invisible.** Once a category hits max, it should disappear from the agent's view entirely.
3. **Refactors can regress scores.** Extracting shared code temporarily increases file count and cross-file references. The scorer should understand "refactor in progress" patterns.
4. **Test coverage is the highest-ROI fix for most projects.** It's usually the biggest category (25pts) and easiest to improve mechanically. Should be prioritized by default.
