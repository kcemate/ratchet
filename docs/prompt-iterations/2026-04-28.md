# Prompt Iteration: 2026-04-28

## Agent: Shell Agent (buildIssuePlanPrompt)

## Failure Pattern Addressed
Agents making syntactically invalid or build-breaking changes despite following SEARCH/REPLACE format correctly, leading to build failures and rolled-back clicks with no score improvement.

**Evidence from F1 Worker Logs (April 1, 2026):**
- fastify-torque.log: `reason":"build failed"` (indicating agent's changes caused build to break)
- express-torque.log: `NO DELTA: express — click landed but score unchanged` (click succeeded but no improvement)
- got-torque.log: `NO DELTA: got — click landed but score unchanged` (click succeeded but no improvement)
- trpc-torque.log: `NO DELTA: trpc — click landed but score unchanged` (click succeeded but no improvement)

While hono showed improvement (+1), most agents were landing clicks that either failed to build or didn't improve the score, suggesting the changes were either breaking or too trivial/tangential.

## Root Cause Analysis
The prompt already included strong constraints against over-engineering (`- Do NOT restructure...`, `- Do NOT add new dependencies...`), but lacked explicit guidance on:
1. Ensuring replacement code is syntactically valid
2. Making changes that directly address the reported issue
3. Avoiding speculative or preventive changes that aren't requested

Agents were technically complying with the SEARCH/REPLACE format but sometimes:
- Made changes that introduced syntax errors (missing brackets, semicolons, etc.)
- Made changes that were technically correct but didn't address the core issue affecting the score
- Made "preventive improvements" that weren't requested and potentially broke existing functionality

## Improvement Applied
Enhanced `buildIssuePlanPrompt` in `/Users/giovanni/Projects/ratchet/src/core/agents/shell.ts` with guidance for making syntactically valid, targeted changes:

### Added to SEARCH BLOCK RULES:
```
- Ensure your replacement code is syntactically valid and matches the language's grammar
- Make minimal, direct changes that fix the reported issue
- Avoid speculative changes or "preventive improvements" that aren't requested
```

This builds upon previous iterations:
- **April 26th**: Added TERMINAL RULES preventing extra characters after REPLACE: (fixed torque exit code 2)
- **April 27th**: Enhanced SEARCH BLOCK RULES with whitespace sensitivity and verification techniques (fixed "Search text not found" errors)

## Expected Outcome
This change should reduce build failures and trivial changes by:
1. Explicitly reminding agents to verify syntactic validity of their replacements
2. Encouraging focus on the reported issue rather than speculative improvements
3. Discouraging overextension beyond what's necessary to fix the problem
4. Complementing existing constraints against restructuring and dependency changes
